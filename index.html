The PoC Challenge: Access to Real Data

Robust AI requires high-quality data. However, accessing sensitive, production-level application data for an initial proof-of-concept presents significant data governance and privacy hurdles.

The Solution: A Methodological Approach

To overcome this, we created a high-fidelity synthetic dataset. This approach allows us to prove the viability of the AI methodologies before engaging with complex data access reviews, de-risking the project entirely.

Methodology: Persona-Driven Synthetic Data

The data was not random. It was generated based on four hypothesised "personas" of underserved applicants.

Stable Gig-Worker

High income, short trading history.

Nearly-There FTB

Good income, high LTV.

Minor Credit Blip

High income, fair credit score.

Slightly Overextended

Good income, high DTI.

This method ensures our mock data has realistic, built-in patterns. The success of the PoC is then measured by the AI's ability to rediscover these personas without any hints, proving its effectiveness.

Auditor Agent: What It Does

The agent follows a dynamic, three-step process for each audit.

1. Fetch & Understand

The agent is given an Applicant ID. Its first action is to use the `get_application_data` tool to retrieve the applicant's details from the dataset, forming its initial understanding of the case.

2. Analyse & Verify

The agent's reasoning engine plans its next steps. It uses the `compliance_checker` tool to ask specific questions (e.g., "What is the policy for a self-employed applicant?") against the lender's PDF knowledge base, verifying key facts like credit score and LTV.

3. Report

Once all checks are complete, the agent synthesises its findings into the structured Markdown report, providing clear pass/fail statuses and a final recommendation, completing its task.

Auditor Agent: Architecture & Rationale

Why an Agentic Workflow?

A simple script fails when faced with ambiguity. An agent, using a ReAct (Reasoning and Acting) framework, can dynamically plan and execute a multi-step process. It can handle unexpected data, decide which tool to use next, and even self-correct, making it a robust choice for a complex task like auditing.

Tech: LangChain Agents, GPT-4-Turbo, Custom Tools

Why RAG over Fine-Tuning?

For compliance, factual accuracy is non-negotiable. Fine-tuning teaches a model a *skill*, but risks factual "hallucination." RAG grounds the agent in a verifiable knowledge source. This approach is easier to update (just swap the PDF), prevents hallucination, and allows for source citations: all critical for an auditable system.

Tech: FAISS, Sentence Transformers, PyPDF, LangChain

Auditor Agent: Real-World Implementation Challenges

While our PoC demonstrates the core functionality, production deployment requires addressing several critical considerations.

Technical Architecture Challenges

  • RAG System Scalability: FAISS index needs to handle multiple lender policies simultaneously
  • Tool Integration: Replace CSV reader with secure database connections and API integrations
  • Model Performance: GPT-4-Turbo costs and latency for high-volume processing
  • Error Handling: Robust fallback mechanisms when tools fail or return unexpected results

Tech: FastAPI, PostgreSQL, Redis, Docker, Kubernetes

Compliance & Auditability

  • Decision Traceability: Every audit decision must be fully explainable and auditable
  • Policy Versioning: Handle updates to lender policies without breaking existing audits
  • Regulatory Compliance: Ensure the system meets FCA requirements for automated decision-making
  • Human Oversight: Build-in escalation paths for complex or edge cases

Operational Considerations

  • Processing Time: Current PoC takes ~30-35 seconds; production needs quicker responses or good visual feedback for the user
  • Concurrent Users: Handle multiple brokers running audits simultaneously
  • Data Security: Encrypt sensitive application data in transit and at rest
  • Monitoring: Real-time alerts for system failures or unusual audit patterns

Tech: Prometheus, Grafana, ELK Stack, OAuth2

Business Integration

  • Workflow Integration: Seamlessly embed into existing broker platforms and CRM systems
  • User Experience: Design intuitive interfaces that don't disrupt existing workflows
  • Training & Adoption: Comprehensive training programs for brokers and underwriters
  • Performance Metrics: Track audit accuracy, time savings, and user satisfaction

Market Finder: What It Does

The dashboard is the final output of a three-stage data science pipeline.

1. Pre-process

The raw data, with its mix of text and numbers on different scales, is cleaned. Categorical data is encoded and all numerical features are scaled to ensure no single feature unfairly dominates the analysis.

2. Cluster

The K-Means algorithm is applied to the high-dimensional processed data. It mathematically groups the applicants into four distinct clusters based on their shared characteristics, assigning a segment label to each applicant.

3. Visualise

The UMAP algorithm creates a 2D "map" of the data. This map is then plotted using Plotly, with each applicant's colour determined by the cluster label found in the previous step, revealing the hidden market structure.

Market Finder: Methodology & Rationale

Why Unsupervised Clustering?

The goal is to find *unknown* market gaps. We don't have pre-existing labels for these segments, making supervised learning impossible. Unsupervised learning (K-Means) is the correct approach because it allows the algorithm to discover the natural structure and patterns within the data itself, letting the "unknown unknowns" emerge.

Tech: scikit-learn, pandas, NumPy

Why UMAP for Visualisation?

While PCA is fast, it's linear and misses complex relationships. t-SNE is good at showing local clusters but distorts global structure. UMAP was chosen as it provides a superior balance, preserving both the local neighbourhood of points and the overall global structure of the data, resulting in a more meaningful and trustworthy visualisation.

Tech: UMAP, Plotly, Streamlit

Real-World Considerations: Determining Valid Outcomes

While our PoC successfully identified four distinct clusters, real-world implementation requires additional validation and refinement.

How We Determine Optimal Clusters

Our PoC uses two complementary methods:

  • Elbow Method: Uses WCSS (Within-Cluster Sum of Squares) to find the point of diminishing returns
  • Silhouette Analysis: Measures cluster separation quality (range: -1 to 1, higher is better)

Tech: KneeLocator library, sklearn.metrics.silhouette_score

Data Preprocessing Pipeline

Before clustering, we standardize the data:

  • Numerical features: StandardScaler for Income, CreditScore, LTV, DebtToIncomeRatio
  • Categorical features: OneHotEncoder for EmploymentStatus

Tech: sklearn.preprocessing, ColumnTransformer

Engineering Roadmap: From PoC to Production

These PoCs are built on scalable principles. Here is a high-level roadmap.

Auditor Agent Service

  • Microservice: Wrap the agent in a FastAPI service for a robust API.
  • Deployment: Containerise with Docker and deploy on a scalable platform like Kubernetes (EKS/GKE).
  • Data Integration: Replace CSV reader with secure connections to production databases and lender APIs.
  • CI/CD: Implement a CI/CD pipeline for automated testing and deployment of agent and tool updates.

Market Insight Platform

  • Data Pipeline: Use Airflow or a similar orchestrator to run the ML pipeline on anonymised data nightly.
  • Data Warehouse: Store processed data and insights in a scalable warehouse like Snowflake or BigQuery.
  • API Layer: Expose insights via a secure API for internal dashboards or as a new data product for lenders.
  • MLOps: Implement model monitoring to detect data drift and retrain the clustering model periodically.
